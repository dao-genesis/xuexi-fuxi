## 三 · PDF 处理 · 开源资源全图

> 本工作区 `pdf_提文.py` 用 PyMuPDF 直取嵌图（PPT→PDF 内含原 jpg/png），
> 无文字层时不独装 OCR，而让 LLM 看图顺手录文（`章占位填充任务`）。
> 下表为天下此领域之器，取之作备选、增强、对比。

### A. 本工作区之底层

| 库 | 链接 | 所司 | 工作区用法 |
|---|---|---|---|
| **PyMuPDF** (fitz) | https://github.com/pymupdf/PyMuPDF | PDF/图像高性能处理·直取嵌图·渲染页面·AGPL | `pdf_提文.py` 核心·`page.get_images()` 直取 jpg/png |

**PyMuPDF 关键 API（已用或可用）**：

```python
import pymupdf  # 亦可 import fitz
doc = pymupdf.open("xxx.pdf")
for page in doc:
    # 直取嵌图（PPT→PDF 之原 jpg/png，比渲染快 10×）
    for img in page.get_images(full=True):
        xref = img[0]
        base = doc.extract_image(xref)
        # base["image"] = 字节；base["ext"] = "jpg"/"png"

    # 渲染为高分辨率图像（扫描件 PDF 用此路）
    pix = page.get_pixmap(dpi=150)
    pix.save(f"page_{page.number:03d}.jpg")

    # 取文字层（本工作区雨课堂 PDF 无文字层，此路为空）
    text = page.get_text("text")

    # 新 API：直接出 Markdown（LLM-ready，1.24.0+）
    md = page.get_text("markdown")
```

### B. 同类之器 · 横向比较

| 工具 | 链接 | 技术栈 | 最适场景 | 与本工作区关系 |
|---|---|---|---|---|
| **MinerU** | https://github.com/opendatalab/MinerU | VLM+OCR 双引擎·109 语言·PDF/DOCX/PPTX/图像→Markdown/JSON·MCP Server | 有文字层的复杂 PDF·含表格/公式 | 本工作区雨课堂 PDF 已是图像合成·**MinerU 可作扫描件 PDF 的备选**；其 pipeline 模式无幻觉且可 CPU 运行 |
| **Marker** | https://github.com/datalab-to/marker | surya OCR + 深度学习布局检测 → Markdown/JSON·可选 LLM 增强 | 书籍/论文结构 PDF·多列/公式 | `--use_llm` 模式与本工作区「LLM 顺手录文」思路相通；可作备选 OCR 管线 |
| **Docling** (IBM) | https://arxiv.org/abs/2501.17887 | PDF+Office→Markdown/JSON·AI 布局理解·LangChain/LlamaIndex 集成 | 企业文档·需语义结构 | 已集成 Langflow；可作 RAG 前处理 |
| **surya** | https://github.com/VikParuchuri/surya | 多语言 OCR + 布局检测·Marker 底层 | 中文等多语言 OCR | Marker 背后·中文支持好 |
| **MarkItDown** (Microsoft) | https://github.com/microsoft/markitdown | 轻量级多格式→Markdown·零模型依赖 | 简单 PDF/Office/HTML 转换 | 若 PDF 有文字层可直接用；比 MinerU 轻 |

**本工作区选型小结（「去彼取此」之道）**：

```
本工作区雨课堂 PDF = 图像合成 PDF（无文字层）
→ PyMuPDF 直取嵌图 = 最快（比渲染快 10×）✓ 已用
→ LLM 顺手录文 = 省 OCR 装包·token 同费·一关二事 ✓ 已用

若遇到扫描件/有文字层 PDF：
→ MinerU（pipeline 模式）= 无幻觉·CPU 可跑·中文好
→ Marker（--use_llm）= 精度高·需 GPU
→ surya（surya_ocr）= 轻量多语言 OCR
```

## 七 · 可立即行动之整合清单

> *为之于其未有也，治之于其未乱也。*

### 优先级 ★★★（直接增益·无需大改）

1. **`.env` 补全 vision model 配置**：按第四节 B 表选型，配好 `LLM_VISION_*`（DeepSeek-VL2 或 Qwen-VL-Max），大幅提升道喂质量。

2. **参考 flosacca/yuketang-dl 补全课件回放提取**：当前 `extractors.py` 已有视频，若有课件录播 replay 需求可参考此项目的回放 URL 解析路径。

3. **参考 zly2006/yuketang-auto-answer 补全 exercises_api**：其 Chrome 扩展对 HTTP 获取 PPT+答案、WebSocket 实时通讯有详尽分析，可作 `api.py:ExercisesAPI` 的答案解密参考（`decrypt_problem.py` 思路）。

4. **参考 heyblackC 心跳包实现完善视频心跳**：`POST /video-log/heartbeat/` 的 payload 结构，`yuketangHelper/videoHelper.py` 已实现。

### 优先级 ★★（中期增益·需一定改造）

5. **MinerU 作扫描件 PDF 备选管线**：若遇到非雨课堂的纯扫描 PDF（如图书馆资料），`pip install mineru` 后一行命令得 Markdown；可封装进 `pdf_提文.py` 作第二路径。

6. **embedded_text 落地后对接 AnythingLLM**：道喂完成后，`_doc.json:embedded_text` 即为全课文字库。导出为标准文档格式，导入 AnythingLLM，主公可用自然语言检索「环境法学第三章提到的什么」。

7. **参考 MuWinds/openai_ask.py 增强 `_道说_提示库.py` 的模拟卷 schema**：其对选择/填空/判断三题型的 JSON schema 设计成熟，可参考优化 `模拟卷任务` 的结构化返回格式。

### 优先级 ★（远期/可选）

8. **LiteLLM 替代 `_llm_道.py`**：若将来需要多 LLM 路由/负载均衡/成本统计，可迁移至 LiteLLM；迁移成本低（接口相似）。当前自研版零依赖·够用。

9. **surya 作本地 OCR 补充**：`pip install surya-ocr`·无 GPU 可跑·中文好。若主公有大量非雨课堂扫描 PDF，可作 `embedded_text` 的纯本地填充路径（不花 API 钱）。

10. **teng-lin/notebooklm-py 对接 NotebookLM**：将 `_最终复习资料.md` 上传 NotebookLM，一键生成播客式音频摘要，解放双手。
